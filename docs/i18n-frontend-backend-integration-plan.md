# Frontend-Backend i18n Integration Audit

## Scope

- This audit only covers backend endpoints that are currently consumed by `PCL.Frontend.Avalonia`.
- I list leaf endpoints that originate or carry user-visible text into the frontend. Pure aggregators such as `LauncherFrontendShellService.BuildPlan(...)` are not repeated as separate rows.
- Frontend-only Chinese literals are out of scope here.
- Prompt transport is centralized by `PCL.Core.Backend/App/Essentials/LauncherFrontendPromptService.cs:11-139,235-246`, and those prompt DTOs are rendered by `PCL.Frontend.Avalonia/ViewModels/FrontendShellViewModel.Prompts.cs:20-24,147-169,1023-1031,1048-1051`. To avoid duplication, prompt sources are listed below at their origin.
- I excluded backend APIs that are not currently referenced by the Avalonia frontend, such as `LauncherUpdateLogService` and `MinecraftCrashResponseWorkflowService`.

## Integration Rule

- The frontend is the only layer that can resolve localization and render localized text.
- Use enums plus structured parameters when the backend is sending domain state, UI structure, action kinds, or other finite state.
- Use content keys plus typed parameters when the backend is selecting a message-bearing copy variant for the frontend to render.
- Do not choose transport format by widget type such as dialog, popup, toast, or page. Choose it by payload semantics.
- Raw text is allowed only for non-localizable diagnostics or external error/detail payloads. Raw text should live in dedicated fields and should not be mixed with localized launcher chrome in the same field.

## Practical Takeaways

1. The biggest i18n blockers are the backend-owned navigation and page-content builders. They should move toward structured state, not just keyed strings.
2. Prompt and notification payloads should not be forced into enum-only modeling. Content keys plus typed params are the better fit when the backend is selecting a copy variant.
3. The task system is still a major text pipe. `TaskState` is already enum-based, but task title, status, and progress text remain largely freeform.
4. Raw text should be treated as a narrow exception path for diagnostics and external error details, not as a general integration pattern.
